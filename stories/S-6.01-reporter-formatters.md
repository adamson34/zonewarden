# S-6.01: Reporter — JSON + Text + Mermaid Formatters

## Narrative
- **As a** zonewarden user
- **I want to** emit machine-readable JSON, human-readable text, and Mermaid diagram output from a `ConformanceResult`
- **So that** I can integrate the tool into CI pipelines, review results visually, and generate topology diagrams with violations highlighted

## Acceptance Criteria

### AC-001 (traces to BC-1.06.002 postcondition — JSON schema compliance)
`reporter::emit_json(result)` produces JSON that passes validation against the JSON schema in `interface-definitions.md`. All required fields are present: `schema_version`, `policy_digest`, `total_flows`, `intra_zone`, `allowed`, `no_matching_conduit`, `wrong_direction`, `multicast_exempt`, `idmz_bypasses`, `distinct_violating_flows`, `external_endpoints`, `skipped`, `warnings`, `violations`.
- **Test:** `test_BC_1_06_002_json_schema_required_fields_present()`

### AC-002 (traces to BC-1.06.002 postcondition — violation fields)
Each violation in the JSON `violations` array has required fields: `flow_index`, `src_zone`, `dst_zone`, `kind` ∈ {`NoMatchingConduit`, `WrongDirection`, `IdmzBypass`}, `severity` ∈ {`Established`, `Attempted`}, `explanation`, `src_ip`, `dst_ip`, `proto`, `service_source`.
- **Test:** `test_BC_1_06_002_violation_required_fields_present()`

### AC-003 (traces to BC-1.06.002 postcondition — schema_version field)
The JSON `schema_version` field is `"1.0"`.
- **Test:** `test_BC_1_06_002_schema_version_is_1_0()`

### AC-004 (traces to BC-1.06.003 postcondition — text report)
`reporter::emit_text(result)` produces a human-readable report that includes: summary line (total_flows, violations count), per-violation lines with src_zone, dst_zone, kind, severity, and `service_source` provenance. The text report surfaces heuristic inferences visibly.
- **Test:** `test_BC_1_06_003_text_report_includes_summary_and_violations()`

### AC-005 (traces to BC-1.06.003 postcondition — service_source in text)
The text report explicitly labels `service_source: PortHeuristic` violations (e.g., appends `[heuristic]` or similar marker). DI-008 — heuristics are always visibly flagged.
- **Test:** `test_BC_1_06_003_text_report_flags_heuristic_service_source()`

### AC-006 (traces to BC-1.06.004 postcondition — Mermaid diagram)
`reporter::emit_mermaid(result, policy)` produces a valid Mermaid `graph LR` diagram with zones as nodes (labeled with zone ID and Purdue level) and conduits as edges. Zones with violations are annotated with `:::violation` style.
- **Test:** `test_BC_1_06_004_mermaid_has_zone_nodes_and_conduit_edges()`

### AC-007 (traces to BC-1.06.004 postcondition — violations highlighted)
Zones involved in at least one violation (NoMatchingConduit or IdmzBypass) appear with a `:::violation` CSS class in the Mermaid output.
- **Test:** `test_BC_1_06_004_violated_zones_highlighted()`

### AC-008 (traces to BC-1.06.004 invariant — Mermaid deterministic)
Mermaid node declarations are sorted by zone ID (lexicographic). The output is deterministic for identical inputs.
- **Test:** `test_BC_1_06_004_mermaid_nodes_sorted_deterministic()`

### AC-009 (traces to BC-1.06.002 invariant — policy_digest in JSON)
The JSON `policy_digest` field is a lowercase 64-character hex string (from `ConformanceResult::policy_digest`).
- **Test:** `test_BC_1_06_002_policy_digest_format_valid()`

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Zero violations | JSON `violations: []`; text shows "No violations"; Mermaid has no `:::violation` nodes |
| EC-002 | IdmzBypass violation (kind = `IdmzBypass`) | JSON kind = `"IdmzBypass"`; text labels it; Mermaid highlights zones |
| EC-003 | Violation with `service: None` | JSON omits `"service"` key; text omits service info |
| EC-004 | Very long zone ID in Mermaid | No truncation; full ID used |
| EC-005 | Policy with no conduits | Mermaid shows zones only, no edges |
