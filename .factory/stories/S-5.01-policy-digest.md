# S-5.01: Policy Digest â€” Canonical JSON + SHA-256

## Narrative
- **As a** zonewarden user
- **I want to** have each run produce a stable `policy_digest` field (SHA-256 of canonical policy JSON)
- **So that** audit reports are uniquely tied to the exact policy version that produced them, enabling reproducible evidence chains

## Acceptance Criteria

### AC-001 (traces to BC-1.05.003 postcondition 1 â€” stable digest)
`digest::compute(policy)` returns the same lowercase 64-character hex SHA-256 string for structurally-equal policies, regardless of the YAML source order they were loaded from.
- **Test:** `test_BC_1_05_003_structurally_equal_policies_same_digest()` (proptest round-trip)

### AC-002 (traces to BC-1.05.003 postcondition 2 â€” canonical order)
The canonical JSON has zones sorted by `id` (lexicographic UTF-8) and conduits sorted by `(from_zone, to_zone, proto_token, direction_token, ports_repr)`.
- **Test:** `test_BC_1_05_003_canonical_sort_order()`

### AC-003 (traces to BC-1.05.003 postcondition 3 â€” members sorted)
Zone members are sorted by their CIDR string representation in the canonical output.
- **Test:** `test_BC_1_05_003_zone_members_sorted()`

### AC-004 (traces to BC-1.05.003 postcondition 4 â€” PortSet canonical)
PortSet values are in canonical form (DI-020) before serialization: `[502, 500-502]` becomes `[500-502]`.
- **Test:** `test_BC_1_05_003_portset_canonical_in_digest()`

### AC-005 (traces to BC-1.05.003 postcondition 5 â€” None fields omitted)
`Option::None` fields are omitted entirely from the JSON (not serialized as `null`). A zone without `sl_t` produces JSON without a `"sl_t"` key.
- **Test:** `test_BC_1_05_003_none_fields_omitted()`

### AC-006 (traces to BC-1.05.003 postcondition 6 â€” lowercase hex output)
The digest is a lowercase 64-character hex string (matching `^[0-9a-f]{64}$`).
- **Test:** `test_BC_1_05_003_digest_format_lowercase_hex()`

### AC-007 (traces to BC-1.05.003 postcondition â€” different policy â†’ different digest)
Two policies that differ by even one field produce different digests. Adding a zone or changing a CIDR changes the digest.
- **Test:** `test_BC_1_05_003_different_policies_different_digests()`

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Empty policy (no zones, no conduits) | Deterministic digest of `{}` canonical form |
| EC-002 | Two policies differing only in YAML comment | Same digest (comments not parsed) |
| EC-003 | Zone `sl_t: None` vs zone without `sl_t` field | Same digest (None omitted) |
| EC-004 | Conduit `ports: any` vs `PortSet::Any` | Both serialize as `"any"` in canonical JSON |
| EC-005 | Policy with duplicate conduits (same tuple) | Deduplicated before serialization (ADR-004 step 3) |
