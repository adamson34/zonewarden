# S-5.03: Aggregator â€” Deterministic Sort + Empty-Input Case

## Narrative
- **As a** zonewarden reporter
- **I want to** receive violations sorted in a deterministic total-order
- **So that** every run with identical inputs produces byte-identical output, making CI diffing and evidence chain comparison reliable

## Acceptance Criteria

### AC-001 (traces to BC-1.05.002 postcondition â€” total-order sort)
After aggregation, `ConformanceResult::violations` is sorted by the total-order key: `(ts, src_ip, src_port, dst_ip, dst_port, proto, flow_index)` in ascending order. Two runs with the same input produce violations in the same order.
- **Test:** `test_BC_1_05_002_violations_sorted_total_order()`

### AC-002 (traces to BC-1.05.002 postcondition â€” tiebreaker is flow_index)
When two violations have identical `(ts, src_ip, src_port, dst_ip, dst_port, proto)`, they are ordered by `flow_index` ascending. This is the final tiebreaker ensuring a strict total order.
- **Test:** `test_BC_1_05_002_flow_index_tiebreaker()`

### AC-003 (traces to BC-1.05.002 invariant â€” sort is stable-equivalent for equal inputs)
Given two structurally-identical input sets, the sort produces identical output. The sort must be deterministic (not insertion-order dependent).
- **Test:** `test_BC_1_05_002_sort_deterministic_proptest()` (proptest)

### AC-004 (traces to BC-1.05.002 â€” IP comparison order)
IP addresses are compared as byte sequences (IPv4 before IPv6; within IPv4/IPv6 by numeric value). Standard `IpAddr` `PartialOrd` implementation is used.
- **Test:** `test_BC_1_05_002_ip_comparison_order()`

### AC-005 (traces to BC-1.05.005 postcondition â€” empty input zero result)
The empty ConformanceResult (from S-5.02) passes through the sort step unchanged: `violations.is_empty()`, all counts zero.
- **Test:** `test_BC_1_05_005_empty_result_unchanged_by_sort()`
