# S-1.03: Policy Semantic Validation

## Narrative
- **As a** zonewarden user
- **I want to** have the policy validator reject all structurally-invalid policies with precise diagnostics before any flows are processed
- **So that** the pipeline never operates on a corrupt or ambiguous policy, and error messages guide me to fix the exact problem

## Acceptance Criteria

### AC-001 (traces to BC-1.01.004 postcondition 1)
`validator::validate` returns `Err(E-POL-005)` when any two zones share the same `id`. The error message names the duplicate id.
- **Test:** `test_BC_1_01_004_duplicate_zone_id_rejected()`

### AC-002 (traces to BC-1.01.004 postcondition 2)
`validator::validate` returns `Err(E-POL-006)` when a conduit references a `from_zone` or `to_zone` that does not exist as a declared zone id (and is not the reserved `EXTERNAL` zone identifier).
- **Test:** `test_BC_1_01_004_conduit_unknown_zone_rejected()`

### AC-003 (traces to BC-1.01.005 postcondition — tie rejection)
`validator::validate` returns `Err(E-POL-007)` when two zones declare overlapping prefixes of the same length and same address family (e.g., two zones both declaring `10.0.1.0/24`). The error names both conflicting zones.
- **Test:** `test_BC_1_01_005_equal_prefix_tie_rejected()`

### AC-004 (traces to BC-1.01.006 postcondition — /0 rejection)
`validator::validate` returns `Err(E-POL-008)` when any zone member is `0.0.0.0/0` or `::/0`. The error names the zone and the offending CIDR.
- **Test:** `test_BC_1_01_006_catch_all_cidr_rejected()`

### AC-005 (traces to BC-1.01.007 postcondition — unrecognized token rejection)
`validator::validate` returns `Err(E-POL-003)` when a conduit contains an unrecognized `direction` or `proto` token. No permissive default is applied. (Note: structural token parsing is handled at load time in S-1.02; this AC covers semantic validation of the enum set.)
- **Test:** `test_BC_1_01_007_invalid_token_rejected()`

### AC-006 (traces to BC-1.01.008 postcondition — warn not error)
When a zone has zero `members`, `validator::validate` succeeds (returns `Ok(ValidatedPolicy)`) but includes a warning string `"zone '<id>' has no members"` in the returned `ValidatedPolicy::warnings` vec.
- **Test:** `test_BC_1_01_008_zero_members_warn_not_error()`

### AC-007 (traces to BC-1.01.004 invariant — EXTERNAL reserved)
The string `"EXTERNAL"` is reserved and MUST NOT appear as a declared zone `id`. If it does, `validator::validate` returns `Err(E-POL-005)` (duplicate/reserved id).
- **Test:** `test_BC_1_01_004_external_reserved_id_rejected()`

### AC-008 (traces to BC-1.01.005 — OQ-004 short-prefix warning)
A zone member with prefix length < 8 (e.g., `10.0.0.0/7`) is accepted but triggers a warning: `"zone '<id>': member '<cidr>' has very short prefix (< /8); may partially shadow EXTERNAL"`. This is per OQ-004 resolution.
- **Test:** `test_BC_1_01_004_short_prefix_warning_emitted()`

### AC-009 (traces to BC-1.01.004 postcondition — ValidatedPolicy output)
`validator::validate` returns `Ok(ValidatedPolicy)` that contains the sorted prefix index (a `Vec<(IpNet, ZoneId)>` sorted descending by prefix length) ready for use by `resolver::resolve`.
- **Test:** `test_BC_1_01_004_validated_policy_contains_sorted_index()`

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Conduit `from_zone: "EXTERNAL"` | Valid (EXTERNAL is a reserved zone id, usable in conduits) |
| EC-002 | Zone id `"EXTERNAL"` declared in policy | `Err(E-POL-005)` — reserved id |
| EC-003 | Two zones: `10.0.0.0/24` and `10.0.0.0/16` (different lengths) | Valid — different prefix lengths, no tie |
| EC-004 | Two zones: both `10.0.0.0/24` | `Err(E-POL-007)` — equal-prefix tie |
| EC-005 | Zone with member `0.0.0.0/0` | `Err(E-POL-008)` |
| EC-006 | Zone with member `::/0` | `Err(E-POL-008)` |
| EC-007 | Zone with `members: []` | `Ok` with warning in `warnings` vec |
| EC-008 | Zone with member `10.0.0.0/7` (prefix < 8) | `Ok` with short-prefix warning (OQ-004) |
| EC-009 | Policy with zero zones | `Ok` — vacuously valid (no conduits either) |
| EC-010 | Policy with zones but no conduits | `Ok` — deny-all is valid policy |

## Architecture Compliance Rules

| Rule | Source | Enforcement |
|------|--------|-------------|
| `validator` is pure — no I/O, no network | ADR-002, BC-1.06.007 | No `std::fs` or `std::net` imports; Kani feasibility |
| `EXTERNAL` is a reserved zone identifier | BC-1.01.004, SS-02-data-model.md | `const EXTERNAL_ZONE_ID: &str = "EXTERNAL"` in types.rs |
| `PrefixIndex` sorted descending by prefix length | ADR-003, BC-1.03.001 | `sort_by(|a, b| b.0.prefix_len().cmp(&a.0.prefix_len()))` |
| All PolicyError variants map to exit 2 | BC-1.06.001, ADR-006 | Established in S-1.02 error type hierarchy |
| Warnings returned in `ValidatedPolicy::warnings`, never printed inside `validator` | SS-01-pipeline.md purity | No `eprintln!` in `validator.rs` |
