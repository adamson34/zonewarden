# S-2.02: Service Inference + Ingest Cap

## Narrative
- **As a** zonewarden pipeline
- **I want to** infer the application-layer service for each flow from its destination port and protocol, and enforce a configurable ingest cap
- **So that** the report surfaces honest OT protocol attribution (with provenance) and the tool never silently processes an unbounded number of flows

## Acceptance Criteria

### AC-001 (traces to BC-1.02.004 postcondition 1)
For a TCP flow to port 502, `service` is `Some("Modbus")` and `service_source` is `ServiceSource::PortHeuristic`.
- **Test:** `test_BC_1_02_004_port_502_tcp_infers_modbus()`

### AC-002 (traces to BC-1.02.004 postcondition 1)
For a TCP flow to port 44818, `service` is `Some("EtherNet/IP")` and `service_source` is `ServiceSource::PortHeuristic`.
- **Test:** `test_BC_1_02_004_port_44818_tcp_infers_ethernet_ip()`

### AC-003 (traces to BC-1.02.004 postcondition 1)
For a UDP flow to port 20000, `service` is `Some("DNP3")` and `service_source` is `ServiceSource::PortHeuristic`. The canonical port/proto table covers at minimum: Modbus/TCP 502, EtherNet/IP TCP/UDP 44818, DNP3 TCP/UDP 20000, S7comm TCP 102, BACnet UDP 47808, OPC-UA TCP 4840.
- **Test:** `test_BC_1_02_004_canonical_ot_ports_covered()`

### AC-004 (traces to BC-1.02.004 postcondition 2)
For a port not in the canonical table (e.g., TCP 9999), `service` is `None` and `service_source` is `ServiceSource::Unknown`.
- **Test:** `test_BC_1_02_004_unknown_port_produces_unknown_source()`

### AC-005 (traces to BC-1.02.004 invariant â€” service_source always set)
`service_source` is never absent on any produced `Flow`. ICMP flows (portless) have `service: None` and `service_source: Unknown`.
- **Test:** `test_BC_1_02_004_service_source_always_present()`

### AC-006 (traces to BC-1.02.006 postcondition)
When the number of successfully-normalized flows reaches `max_flows` (default 1,000,000 or configured via `--max-flows`), the adapter returns `Err(SysError::CapExceeded { max, count })`. The pipeline emits no ConformanceResult, no partial report, and exits 2 with a diagnostic naming the cap and the count at abort (OQ-006 resolution).
- **Test:** `test_BC_1_02_006_cap_breach_aborts_with_exit_2()`

### AC-007 (traces to BC-1.02.006 precondition â€” cap is configurable)
`--max-flows 0` is rejected as a usage error (exit 2, E-SYS-002) before any flows are read.
- **Test:** `test_BC_1_02_006_zero_max_flows_is_usage_error()`

### AC-008 (traces to BC-1.02.006 â€” OQ-006: no partial output on cap breach)
On cap breach, no JSON/text/Mermaid report is written. Exit 2 is the only output (besides the stderr diagnostic).
- **Test:** `test_BC_1_02_006_no_partial_report_on_cap_breach()`
