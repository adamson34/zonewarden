# S-4.03: Classifier â€” IntraZone, Conduit Any-Match, Directionality, Deny-by-Default

## Narrative
- **As a** zonewarden pipeline
- **I want to** assign a deterministic verdict to every resolved flow pair through conduit matching and deny-by-default logic
- **So that** the system correctly allows intra-zone traffic, matches conduits with any-match union semantics, enforces directional constraints, and denies all unmatched flows

## Acceptance Criteria

### AC-001 (traces to BC-1.04.001 postcondition)
`classifier::classify(ctx, flow, pair)` returns `VerdictKind::IntraZone` when `pair.src.zone_id == pair.dst.zone_id`. No conduit evaluation occurs.
- **Test:** `test_BC_1_04_001_same_zone_intra_zone()`

### AC-002 (traces to BC-1.04.002 postcondition â€” deny-by-default)
When no conduit exists between the (src_zone, dst_zone) pair, or no conduit matches the flow's proto/port/direction, `classifier::classify` returns `VerdictKind::NoMatchingConduit`.
- **Test:** `test_BC_1_04_002_no_matching_conduit_verdict()`

### AC-003 (traces to BC-1.04.003 postcondition â€” any-match union)
A flow is `Allowed` if at least one conduit permits it. Given two conduits between the same zone pair â€” one matching TCP/502 and one matching TCP/44818 â€” a TCP/502 flow matches the first conduit and returns `Allowed { conduit_idx: 0 }`.
- **Test:** `test_BC_1_04_003_any_match_first_conduit_permits()`

### AC-004 (traces to BC-1.04.004 postcondition â€” forward directionality)
A `Forward` conduit from ZoneA to ZoneB permits flows initiated from ZoneA â†’ ZoneB. A reverse-initiated flow (ZoneB src, ZoneA dst) against the same `Forward` conduit returns `VerdictKind::WrongDirection`.
- **Test:** `test_BC_1_04_004_forward_conduit_rejects_reverse()`

### AC-005 (traces to BC-1.04.005 postcondition â€” bidirectional)
A `Bidirectional` conduit from ZoneA to ZoneB permits initiation from either direction. A flow from ZoneB â†’ ZoneA matches the bidirectional conduit and returns `Allowed`.
- **Test:** `test_BC_1_04_005_bidirectional_permits_both_directions()`

### AC-006 (traces to BC-1.04.006 postcondition â€” portless protocol)
ICMP and `Other(u8)` flows match only conduits with `PortSet::Any`. A conduit with `ports: [502]` does NOT match an ICMP flow. A conduit with `ports: Any` DOES match an ICMP flow.
- **Test:** `test_BC_1_04_006_icmp_matches_any_portset_only()`

### AC-007 (traces to BC-1.04.009 postcondition â€” severity grading)
The `Verdict` returned for a violation (`NoMatchingConduit` or `WrongDirection`) includes a `Severity` derived from `severity::grade(flow.conn_state)`. A flow with `conn_state: Some(Established)` gets `Severity::Established`.
- **Test:** `test_BC_1_04_009_violation_severity_from_conn_state()`

### AC-008 (traces to BC-1.04.003 â€” WrongDirection wins over NoMatchingConduit)
If the (src_zone, dst_zone) pair has conduits that would match if direction were reversed, but direction is wrong, the verdict is `WrongDirection` (not `NoMatchingConduit`). This preserves the directional failure signal over the generic deny.
- **Test:** `test_BC_1_04_004_wrong_direction_beats_no_match()`

### AC-009 (traces to BC-1.04.001 â€” no IDMZ check for IntraZone)
IntraZone flows do NOT trigger IDMZ check. `idmz::check` is called only for non-intrazone resolved pairs.
- **Test:** `test_BC_1_04_001_intra_zone_skips_idmz_check()`
