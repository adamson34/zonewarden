# S-4.02: IDMZ No-Bypass Truth Table

## Narrative
- **As a** zonewarden security engine
- **I want to** detect any direct OT-to-IT flow that bypasses the Industrial DMZ
- **So that** the headline IEC 62443 security control — the Purdue 3.5 boundary — is enforced and every bypass is surfaced as an additive finding

## Acceptance Criteria

### AC-001 (traces to BC-1.04.007 postcondition 1)
`idmz::check(src_zone, dst_zone, policy)` returns `idmz_bypass = true` when one endpoint is in a managed zone ≤ L3 (L0/L1/L2/L3) and the other is in a managed zone ≥ L4 (L4/L5), and neither is IDMZ (L3.5). Example: src=L2, dst=L4.
- **Test:** `test_BC_1_04_007_ot_to_it_triggers_bypass()`

### AC-002 (traces to BC-1.04.007 postcondition 4 — additive)
`idmz::check` is additive: it returns `idmz_bypass` independently of the conduit verdict. The function does NOT modify or know about `VerdictKind`. A flow can have both `Allowed` verdict AND `idmz_bypass = true`.
- **Test:** `test_BC_1_04_007_bypass_additive_to_conduit_verdict()`

### AC-003 (traces to BC-1.04.008 postcondition — EXTERNAL exclusion)
When either endpoint is the `EXTERNAL` zone (zone_id == "EXTERNAL"), `idmz::check` returns `idmz_bypass = false` regardless of the other endpoint's Purdue level. This is the DI-006 truth table row 3.
- **Test:** `test_BC_1_04_008_external_endpoint_no_bypass()`

### AC-004 (traces to BC-1.04.008 — MulticastBroadcast exclusion)
When the dst endpoint is `MulticastBroadcast`, `idmz::check` returns `idmz_bypass = false` (DI-016; the single exception to verdict-independence per truth table).
- **Test:** `test_BC_1_04_008_multicast_dst_no_bypass()`

### AC-005 (traces to BC-1.04.007 invariant 5 — IDMZ endpoint)
When either endpoint is in an IDMZ (L3.5) zone, `idmz_bypass = false`. The IDMZ zone IS the sanctioned intermediate — flows involving it are compliant.
- **Test:** `test_BC_1_04_007_idmz_endpoint_no_bypass()`

### AC-006 (traces to BC-1.04.007 edge case EC-002 — managed L5)
A declared zone at `purdue_level: L5` (managed, not EXTERNAL sentinel) is treated as a managed ≥ L4 endpoint. Flow from managed L1 to declared L5 zone triggers `idmz_bypass = true`.
- **Test:** `test_BC_1_04_007_declared_l5_is_managed()`

### AC-007 (traces to BC-1.04.007 edge case EC-006 — same side)
Flow from managed L2 to managed L3 (both ≤ L3): `idmz_bypass = false`. Flow from managed L4 to managed L5 (both ≥ L4): `idmz_bypass = false`.
- **Test:** `test_BC_1_04_007_same_side_no_bypass()`

### AC-008 (traces to BC-1.04.007 invariant 2 — EXTERNAL by identity)
EXTERNAL exclusion is by zone identity (`zone_id == EXTERNAL_ZONE_ID`), not by purdue_level value. A declared zone labeled `purdue_level: L5` is NOT excluded.
- **Test:** `test_BC_1_04_007_external_exclusion_by_identity_not_level()`
